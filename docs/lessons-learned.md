\# Lessons Learned



\## 1. Begin with Broad Behavioral Searches



A realistic SOC investigation should not begin by filtering directly for an account already known to be malicious.



The preferred workflow is:



```text

Search by Event ID or behavior

&#x20;       ↓

Identify suspicious entities

&#x20;       ↓

Pivot to focused investigation

```



This approach better reflects real-world triage and reduces confirmation bias.



\## 2. Event Correlation Is More Valuable Than Individual Events



A single event rarely explains the full attack.



The lateral movement investigation became meaningful only after correlating:



```text

4688 on CLIENT01

```



with:



```text

4624 and 4672 on DC01

```



This connected:



\- The source endpoint

\- The initiating user

\- The executed process

\- The destination host

\- The remote account

\- The source IP

\- The authentication method



\## 3. Windows Fields May Not Use the Expected Account Format



Event ID `4728` represented the added member in `MemberName` as a Distinguished Name:



```text

CN=Deployment Service,CN=Users,DC=simaanlab,DC=local

```



It did not display the SamAccountName:



```text

svc.deploy

```



Analysts must understand the structure of Windows Security Event fields before assuming data is missing.



\## 4. Event ID 4672 Can Be Noisy



Event ID `4672` is frequently generated by:



\- SYSTEM

\- LOCAL SERVICE

\- NETWORK SERVICE

\- DWM accounts

\- Administrative services

\- Legitimate privileged users



Noise reduction requires filtering and careful entity mapping.



Incorrect incident grouping can also combine unrelated users and devices into one incident.



\## 5. Ingestion Time Matters



Events may be generated locally before they appear in Microsoft Sentinel.



A rule that searches only a short period based on `TimeGenerated` may miss delayed events.



Analysts should compare:



```text

TimeGenerated

```



with:



```text

ingestion\_time()

```



\## 6. Analytics Rules Require Tuning



A working KQL query does not automatically mean that a scheduled analytics rule will create the expected incident.



The analyst must verify:



\- Rule status

\- Run frequency

\- Lookup window

\- Alert threshold

\- Entity mapping

\- Incident creation

\- Alert grouping

\- Suppression

\- Last run status



\## 7. Saved Queries Reflect Real SOC Operations



SOC analysts do not rewrite every KQL query from the beginning.



Queries are normally:



\- Saved

\- Reused

\- Modified

\- Added to Hunting

\- Converted into Analytics Rules

\- Referenced during investigations



The `queries` directory represents this realistic working method.



\## 8. Containment Must Address Existing Sessions



Disabling an account does not always immediately terminate every existing session.



The lab included:



```cmd

net use \* /delete /y

klist purge

```



to remove active SMB connections and Kerberos tickets.



In production, targeted session cleanup is usually preferable to broad deletion.



\## 9. Response Should Be Documented



Technical containment alone is not enough.



The incident was also:



\- Assigned

\- Commented

\- Classified

\- Tagged

\- Resolved



This creates an auditable investigation record.



\## 10. Troubleshooting Is Part of Detection Engineering



The Azure Monitor Agent issue was not separate from the security project.



Without reliable telemetry, detections and investigations cannot succeed.



Detection engineering therefore includes:



\- Log-source validation

\- Agent health checks

\- DCR verification

\- Ingestion-delay analysis

\- Rule tuning

\- Evidence validation



\## Final Takeaway



The most important lesson from the project is that effective SOC analysis requires more than identifying one suspicious event.



A complete investigation combines:



```text

Telemetry

Detection Logic

Entity Analysis

Correlation

Timeline Reconstruction

Incident Handling

Containment

Remediation

Documentation

```



This end-to-end workflow is what transforms raw Windows logs into a professional security investigation.
